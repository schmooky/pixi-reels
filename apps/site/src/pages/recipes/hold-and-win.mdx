Press Run a few times. every coin that lands stays put on the next respin, and the HUD under the board is driven purely by board events.

A Hold & Win board spins per **cell**, not per column — the engine's atomic spin unit is the reel, so each cell is its own 1×1 `ReelSet`. `HoldAndWinBuilder` (in `examples/shared/`, copy it into your game) owns that grid plus the choreography every H&W game repeats: spin the free cells, lock the hits, reset-or-decrement the counter, detect the full board. The library still doesn't own the mechanic; the board is built entirely on public engine API.

```ts
import { HoldAndWinBuilder } from './shared/HoldAndWinBuilder.ts';

const board = new HoldAndWinBuilder<{ value: number }>()
  .grid(5, 3)
  .cellSize(72, { gap: 4 })
  .symbols(r => r.register('coin', CoinSymbol, COIN_TRIGGER))
  .weights({ coin: 1, empty: 3 })   // coins flash past empties on the strip
  .respins(3)
  .ticker(app.ticker)
  .build();

board.events.on('coin:locked', ({ coin, locked, capacity }) => hud.update(coin, locked, capacity));

// Feature flow — the server stays the only source of truth:
board.enter(trigger.coins);                       // seeds land locked, counter fills
while (true) {
  const round = await server.respin(board.lockedCoins);
  const result = await board.respin(round.hits);  // wave lands, hits lock, counter resolves
  if (result.done) break;                         // anything can animate between rounds
}
```

## Who owns what

1. **The server** decides which cells hit and what each coin carries. A coin is `{ cell, id, data }` — `id` picks the registered art, `data` is an opaque payload the board never reads.
2. **The board** owns choreography: independent per-cell spins with a staggered landing wave, lock bookkeeping, the reset-on-hit counter, `board:full`. It interprets nothing.
3. **The game layer** owns meaning. Adders, doublers, collectors, jackpot tiers are all expressed through the board's three openings — events (every beat fires with the coin payload), `symbolAt(cell)` (the live symbol instance, cast to your own class), and `cellBounds()` / `cellCenter()` (pixel geometry for flights and trails). Collected coins that fly away are removed with `release(cells)`, which frees those cells for future respins — whether anything lands there again is, as always, the server's call.

## Simpler variant: `spin({ holdReels })` on a single ReelSet

The single-coin / column-respin patterns ("hold reels 0 and 4, reroll the middle three") used to require N independent `ReelSet` instances bundled into a row. With the new `holdReels` option that's one `spin()` call:

```ts
// Hold the outer columns; only middle three reroll.
const spin = reelSet.spin({ holdReels: [0, 4] });
reelSet.setResult(serverGrid); // entries at indices 0 and 4 are ignored
await spin;
```

Behaviour:
- Held reels skip START / SPIN / STOP entirely. they don't move at all.
- They count as already-landed for `spin:allLanded`, so the resolver fires once the non-held reels land.
- `setAnticipation([...])` silently filters held indices; `setStopDelays([...])` ignores entries at held indices.
- `SpinResult.symbols` reports the full visible grid (held rows unchanged, non-held rows landed).

When to pick which:
- **Per-cell hold-and-win** (this recipe's main pattern, with coins locked one cell at a time): `HoldAndWinBuilder`'s independent grid is the model — each cell is its own `ReelSet` with `visibleRows(1)`.
- **Single-reel respin / partial reroll** (whole columns held, others spin): use `spin({ holdReels })` on a single `ReelSet`. Spotlight, anticipation, win presenters keep working since everything stays inside one engine.
